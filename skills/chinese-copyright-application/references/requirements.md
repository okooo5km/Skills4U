## 申请材料清单

提交四份材料：

1. **著作权登记申请表** — 软件基本信息、著作权人信息、软硬件环境、功能简介。**直接以 Markdown 文件提交（`.md`），无需编译为 PDF**，申请表本身没有格式硬要求。
2. **源程序** — 前 30 页 + 后 30 页，共 60 页（PDF，由 LaTeX 编译生成）
3. **用户手册** — 详细描述软件使用方法，不少于 10 页，应包含界面截图（PDF）
4. **设计说明书** — 详细描述软件设计思路和实现，不少于 10 页，应包含界面截图（PDF）

> **为什么申请表用 Markdown**：旧版本走 LaTeX→PDF，PDF 文本提取时复制带换行（PDF 文本流的固有问题），审查员录入时容易出错。Markdown 既保留结构又能直接复制，且不受 LaTeX 环境约束，用户没装 LaTeX 也能拿到申请表。

## 源程序文档要求

### 页面规格

- 前 30 页 + 后 30 页 = **正好 60 页**（不能多也不能少）
- 代码连续排列，由 LaTeX 自然分页
- 通过 `10pt` 字号 + `\small` + 窄边距（top=2cm,bottom=2cm,left=1.5cm,right=1.5cm）使每页自然排满约 50 行
- **禁止**每 50 行强制分页（会产生大量空白页）

### 首页信息

源程序文档首页需标注：
- 软件名称 + 版本号
- 著作权人名称
- 生成日期
- 源程序量（行数，必须与申请表一致）
- "本文档含前30页 + 后30页"

### 代码提取策略

1. 按文件优先级排序：入口文件 > 业务逻辑 > 工具函数 > 页面/组件 > 配置
2. 每个文件开头加注释标记：`// ========== path/to/file.js ==========`
3. 文件分隔注释行不计入源程序总量
4. 前 30 页从排序后的第一个文件开始连续取代码
5. 后 30 页从排序后的最后一个文件开始反向取代码
6. 如果代码不足 60 页，全部提供

### 源程序量说明

- 申请表中的「源程序量」= 项目实际总行数
- 源程序文档中展示的代码 ≈ 3000 行（前后各约 1500 行的样本）
- 两个数字不需要相等，但所有文档中引用的总行数必须一致

### 60 页硬约束

- 源程序 PDF **必须正好 60 页**（不能 59、不能 61），唯一例外：项目源代码总量本就不足 60 页，此时全部提供。
- 编译完成后用 `pdfinfo`（或 macOS 上 `mdls -name kMDItemNumberOfPages`）核对页数；若页数不对，调整 `--front-lines` / `--back-lines` 参数微调每页代码量重新生成。
- 默认值：前 30 页目标 1490 行、后 30 页目标 1498 行（合计 2988 行去空行后的代码）。不同字体度量、不同 LaTeX 版本可能产生 ±1 行偏差，需按实测调整。

## 源程序与文档对应规则

> 这是上次（集享 V1.0 软著流水号 2026R11L0626727）被打回补正的第二大原因，必须严格遵守。

**双向对应**：

1. **源代码 → 文档**：源程序鉴别材料（前 30 页 + 后 30 页）中出现的每个文件/模块，在设计说明书 §4 详细设计中**必须有对应小节**。审查员会拿源代码逐项核对设计说明书。
2. **文档 → 源代码**：设计说明书中提到的每个功能模块，**必须能在源程序 60 页内被审查员看到**对应代码。文档里写了一个模块，源代码里没出现，等于"虚构功能"。

**遗留代码必须排除**：

下列文件即使存在于项目中，也不应进入源程序鉴别材料；如果不可避免地进入了，文档里也不应该提及它们：

- `mockData*.js` / `mock*.js` / `fakeData*.js` — 仅供开发期使用的假数据
- `utils/util.js` — 微信小程序默认模板生成的样板（仅含 `formatTime` 等空壳函数）
- `pages/logs/*` — 微信开发者工具默认模板的日志页
- `miniprogram_npm/` — 编译产物，已在 EXCLUDE_DIRS 中
- 任何空文件、占位文件、临时调试代码

工具脚本会扫描出疑似遗留文件并通过 `legacy_candidates` 字段报告，由 Claude 与用户确认后通过 `--exclude-pattern` 排除。

### 内容结构

1. **软件简介** — 软件概述 + 主要特点
2. **功能概述** — 功能列表 + 适用场景
3. **安装与使用说明** — 系统要求 + 进入方式 + 登录说明 + 使用指南入口

### 编写要求

- 语言简洁，步骤清晰
- 每个功能必须配截图
- 不少于 10 页
- 从用户角度描述，避免技术术语

1. **软件概述** — 简介 + 开发背景 + 设计目标

## 版本号一致性核验

> 这是上次（集享 V1.0 软著）被打回补正的**首条意见**：申请表填 V1.0，但项目 `utils/config.js` 里写的是 `version: '1.1.1'`，连截图右下角也是 V1.1.1，被审查员第一眼抓到。

**核心规则**：申请表填的版本号、项目源代码里的版本字符串、运行截图上显示的版本号——三方必须完全一致。

### 常见版本号出现位置

提交前用以下 grep 模板扫一遍项目，把所有命中位置和对应字符串列出来：

```bash
# 扫描项目内的版本号字符串
grep -rEn '"version"\s*:\s*"[0-9]+(\.[0-9]+)+"' . \
  --include='*.js' --include='*.ts' --include='*.json' \
  --exclude-dir=node_modules --exclude-dir=miniprogram_npm

# 扫描页面 footer / about 页里硬编码的版本号
grep -rEn 'V[0-9]+\.[0-9]+|版本.*[0-9]+\.[0-9]+' \
  pages/ components/ src/ 2>/dev/null

# 申请表与文档页眉
grep -n 'V[0-9]\+\.[0-9]\+' copyright-materials/*.md copyright-materials/*.tex
```

需要核对的位置：

- `package.json` 的 `version` 字段
- `utils/config.js` / `src/config.ts` 等配置文件里的 `version` 常量
- 个人中心页/关于页 footer 上硬编码的版本号
- 提交的运行截图（特别是设置页、关于页、启动页）
- 申请表的「版本号」字段
- 用户手册、设计说明书的页眉（`{{软件全称}}V1.0`）
- 源程序文档首页与页眉

**一旦发现不一致，必须停下来让用户裁决「最终申请使用哪个版本号」**，并指引用户在提交前把所有位置统一（包括重新截图）。

## 设计说明书与源代码对应规则

> 详细规则见上一节[源程序与文档对应规则](#源程序与文档对应规则)。本节聚焦设计说明书 §4 的写作结构。

### §4 详细设计的 6 段结构（强制）

每个模块的 `\subsection` 必须按以下 6 段结构展开，缺段会被打回。模板见 [design-doc-template.tex](design-doc-template.tex)。

| 段 | 标题 | 内容 |
|---|---|---|
| 1 | 模块职责 | 一句话定位 + 1–2 句展开 |
| 2 | 输入与输出 | 用户操作、上游数据、跳转参数；渲染结果、下游消费、接口返回 |
| 3 | 关键算法与处理逻辑 | 核心算法、判定规则、必要的伪代码 |
| 4 | 状态与缓存管理 | 本地状态、持久化、失效与刷新策略 |
| 5 | 与其他模块的依赖关系 | 依赖了谁、被谁依赖 |
| 6 | 处理流程 | 编号步骤列表 |

### 模块覆盖双向原则

- 模块小节标题必须与项目实际目录或文件名对应（如 `pages/index`、`utils/api.js`），不要起架空的中文名
- 源代码出现 → 文档必须有；文档提到 → 源代码必须能找到
- 工具/适配器/帮助函数（utils/marketHelper.js、utils/lunar.js 等）也要写小节，不能跳过

## 截图管理

### 截图清单生成

分析项目代码中的页面文件、弹窗组件、模态框，自动列出需要截图的界面。

截图清单格式：

| 编号 | 文件名 | 描述 | 触发方式 | 使用位置 |
|------|--------|------|----------|----------|
| 01 | 图1-首页空状态.png | 首页空状态 | 首次进入首页 | 用户手册§1、设计说明书§4.1 |

### LaTeX 中的截图处理

- 图片统一放在 `images/` 目录下
- 使用 `\graphicspath{{images/}}` 指定图片路径
- 使用自定义 `\screenshot{文件名}{标题}` 宏插入截图
- 编译时如果图片文件不存在会报错，但可以先注释掉 `\screenshot` 行查看文字排版

brew install --cask basictex

**Ubuntu/Debian**：

sudo apt install texlive-xetex texlive-lang-chinese texlive-latex-extra texlive-fonts-recommended

ctex, xecjk, fancyhdr, lastpage, fancyvrb, tabularx, enumitem, float, xcolor, longtable, graphicx, hyperref, array, geometry

```bash

```
